# Research Data Contract

Researchers should add one candidate per file under `trip-data/research-inbox/<agent-or-topic>/`.

Research inbox candidates still use JSON. Canonical trip-planner files that are meant for manual editing live under `trip-data/input/`.

## Editable Inputs

- `viewing-options.csv`: race viewing candidates. Includes a blank `overview` column for your notes.
- `lodging.csv`: one row per lodging stay. Use pipe-separated `stageIds` to link each stay to its itinerary days. Includes blank `overview` and `confirmed` columns.
- `activities.yaml`: planned day activities grouped by itinerary day for easier manual editing. Supports multiline `overview`, YAML lists for `optionIds`, and nested `coordinates`.
- `places.csv`: non-race places to show on the map, such as cafes, restaurants, landmarks, transit anchors, and other trip context. Use pipe-separated `stageIds` and `tags`.
- `transit.yaml`: point-to-point movement rows grouped by itinerary day, with `name`, `overview`, `time`, `start`, `end`, `method`, `link`, and `confirmed`. `link` can be either a single URL string or a YAML object of named links such as `map`, `tickets`, or traveler-specific ticket links.
- `itinerary/*.yaml`: one editable day packet per itinerary day. Use these for day titles, summaries, route labels, map/profile links, route geometry, and day-level resources.

Use either the itinerary id in `day`, such as `trip-jul-07` or `trip-jul-04`, or a trip calendar date in `YYYY-MM-DD` format. Calendar dates are normalized to the matching itinerary id when the app loads the CSV. Pipe-separate list fields like `tags` or `sources`.

`activities.yaml` entries can also include optional `coordinates` and `googleMapsUrl` fields for activities with a meaningful physical location. Activities without those fields continue to work normally and do not appear as map markers.

## Activities YAML Shape

`trip-data/input/activities.yaml` is grouped by stage id:

```yaml
trip-jul-18:
  - id: trip-jul-18-paris-shopping
    type: shopping
    name: Paris Shopping
    summary: Flexible Paris shopping block with department store and luxury boutique options.
    status: needs-details
    startTime: Flexible
    optionIds:
      - paris-galeries-lafayette-haussmann
      - paris-le-bon-marche
    coordinates:
      lat: 48.87
      lng: 2.33
    googleMapsUrl: https://www.google.com/maps/search/?api=1&query=Paris
```

Each activity entry supports:

- `id`: stable unique slug
- `type`: activity category such as `meal`, `ride`, `museum`, `shopping`, or `FYI`
- `name`: display name
- `overview`: optional longer planning note; useful for multiline notes
- `summary`: one-line description used in cards and timeline text
- `status`: optional planning state such as `planned` or `needs-details`
- `startTime`: optional display time
- `endTime`: optional display end time
- `optionIds`: optional YAML list of related option or place ids
- `coordinates.lat` / `coordinates.lng`: optional map coordinates
- `googleMapsUrl`: optional direct map link

## Transit YAML Shape

`trip-data/input/transit.yaml` is grouped by stage id:

```yaml
trip-jul-13:
  - id: transit-2026-07-13-bordeaux-paris
    name: Bordeaux to Paris
    overview: Train from Bordeaux to Paris.
    time: 10:47-12:55
    start: Bordeaux train station
    end: Paris / Grand Hôtel Malher
    method: train
    link:
      map: https://www.google.com/maps/dir/?api=1&origin=Bordeaux
      tickets: https://example.com/tickets
    confirmed: true
```

Each transit entry supports:

- `id`: optional stable unique slug; autogenerated if omitted
- `name`: display name
- `overview`: one-line or multiline planning note
- `time`: display time
- `start`: origin label
- `end`: destination label
- `method`: transport mode such as `train`, `drive`, `flight`, or `taxi/tram`
- `link`: optional directions/booking links, either as a single URL or a key/value object such as `map`, `tickets`, `Scott ticket`, or `Holly ticket`
- `confirmed`: boolean confirmation state

## Day Packet Fields

Each itinerary YAML is a trip-day packet first. Non-race days should omit race fields entirely. Race-focused days can add an optional `raceInfo` object. Do not add itinerary-level workflow fields like `status`, `decisionStatus`, or `priority`:

```json
{
  "id": "trip-jul-04",
  "summary": "Opening team time trial through Barcelona.",
  "dayPlan": "Watch from Montjuic Park per the itinerary.",
  "raceInfo": {
    "startTime": "13:20 local",
    "endTime": "17:35 local",
    "officialUrl": "https://www.letour.fr/en/stage-page",
    "routeMap": {
      "label": "Official stage map",
      "imageUrl": "https://example.com/map.png",
      "url": "https://www.letour.fr/en/stage-page",
      "note": "Official route map or source page"
    },
    "elevationProfile": {
      "label": "Official stage profile",
      "imageUrl": "https://example.com/profile.png",
      "url": "https://www.letour.fr/en/stage-page",
      "note": "Official profile or source page"
    }
  },
  "sources": [
    {
      "title": "Official stage page",
      "url": "https://www.letour.fr/en/stage-page",
      "note": "Race timing, route, and elevation profile"
    }
  ]
}
```

Use local time for race timing and note whether it is official, estimated, or inferred.

For map plotting, use `raceInfo.route` as `[lat, lng]` points. If the source is an official map image or time schedule rather than GPX/GeoJSON, mark the route as approximate in `summary`, `dayPlan`, or a source note.

When official ASO ArcGIS geometry is available, add `raceInfo.officialRoute` and keep `raceInfo.route` only as a fallback:

```json
{
  "officialRoute": {
    "source": "arcgis",
    "layerUrl": "https://services9.arcgis.com/.../FeatureServer/2",
    "where": "Etape=2",
    "note": "Official ASO ArcGIS route geometry from the embedded Tour stage map"
  }
}
```

## Canonical Places CSV

`trip-data/input/places.csv` is the editable source for non-event places that should appear on the map.

Columns:

- `id`: stable unique slug
- `type`: place category such as `coffee`, `food`, `culture`, `bar`, `transit`, or `general_place`
- `name`: display name
- `summary`: short one-line description for cards and popups
- `description`: optional longer planning note
- `stageIds`: pipe-separated itinerary ids such as `trip-jul-05|trip-jul-06`
- `lat`: decimal latitude
- `lng`: decimal longitude
- `googleMapsUrl`: direct Google Maps link
- `tags`: optional pipe-separated metadata tags
- `status`: optional workflow state; defaults to `candidate`
- `confidence`: optional confidence level; defaults to `medium`

## Accepted Types

- `viewing_spot`
- `coffee`
- `food`
- `parking`
- `train_station`
- `lodging`
- `transit`
- `risk_note`

## Accepted Statuses

- `candidate`
- `needs-research`
- `promising`
- `shortlisted`
- `selected`
- `planned`
- `confirmed`
- `rejected`

## Curation Flow

1. Researcher adds candidate JSON under `trip-data/research-inbox/`.
2. Planner review checks source quality and coordinates.
3. Good candidates are promoted into canonical app data.
4. Rejected candidates remain in the inbox with a short reason.
